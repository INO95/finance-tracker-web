# 04. 설계 의도와 트레이드오프

- 한 줄 요약: 이 프로젝트는 "빠른 개인 운영"을 위해 단순한 구조를 택했고, 그 대가로 일부 확장성/표준화 비용을 미래로 미뤘습니다.
- 언제 읽는지: 기능을 추가하기 전, 기존 설계의 기준과 한계를 알고 싶을 때
- 대상 독자: 비전공자, 초급 개발자
- 읽는 시간: 12분
- 선행 문서: `docs/guide/01-system-overview.ko.md`, `docs/guide/03-data-and-storage.ko.md`
- 핵심 용어 3개: 트레이드오프(Trade-off), 기술 부채(Technical Debt), 설계 원칙(Design Principle)
- 코드 근거 경로: `public/index.html`, `src/app.js`, `src/router.js`, `src/services/summaryService.js`, `src/repository/storage.js`

## 3분 요약

- 프론트는 `index.html + assets/js` 모듈 구조로, 진입은 단순하게 유지하면서 유지보수를 개선했습니다.
- 서버는 프레임워크 없이 라우터 중심이라 학습 부담이 낮습니다.
- 저장소는 드라이버 분리로 유연성을 확보했지만 ORM 수준 추상화는 아닙니다.

## 대안 비교

### 접근 A: 현재 방식(단순 구조 우선)

- 프론트: `public/index.html` + `public/assets/js/*` 분리 구조
- 서버: Node 기본 `http` + 수동 라우팅
- 저장: SQLite/JSON 드라이버 선택

장점:

- 의존성 적음
- 코드 흐름이 눈에 잘 들어옴
- 개인 프로젝트 운영 비용 낮음

단점:

- 파일 단위 분리가 약함
- 기능이 커지면 모듈 경계 설계와 테스트가 중요해짐
- 타입 안정성(TypeScript 등) 부재

### 접근 B: 프레임워크 중심 구조

- 프론트: React/Vue + 상태관리
- 서버: Express/Fastify + 계층 분리
- 저장: ORM + 마이그레이션 체계

장점:

- 팀 개발/확장에 유리
- 테스트 및 모듈화 체계 강화

단점:

- 러닝커브와 설정 비용 증가
- 작은 개인 앱에서는 과설계 가능성

선택 이유:

- 이 저장소는 "지금 당장 쓰는 개인 가계 운영"이 우선이라 접근 A가 합리적입니다.

## 구성요소별 설계 의도

### 모듈 분리 프론트

- [사실] UI 마크업은 `public/index.html`, 로직은 `public/assets/js/*`에 분리되어 있습니다.
- [추론] 초기 개발/수정 속도를 최우선으로 둔 결정입니다.

### 서버 라우터 중심 구조

- [사실] `src/router.js`가 검증, 응답, 엔드포인트 분기 대부분을 담당합니다.
- [추론] 디버깅 시 진입점이 분산되지 않게 하려는 의도입니다.

### 저장소 추상화

- [사실] `src/repository/storage.js`가 드라이버를 분기합니다.
- [추론] 데이터 운영 전략이 바뀌어도 API 레이어를 최대한 유지하려는 의도입니다.

### 예산/식비 정책

- [사실] `computeEffectiveFood()`는 식비/총무/환급 패턴을 반영합니다.
- [추론] 단순 지출 합계보다 "내 실제 부담"을 더 중요하게 본 정책 설계입니다.

## 멀티 관점 시뮬레이션

- Architect: 구조가 커질수록 모듈 경계/계약 테스트를 더 엄격히 관리해야 합니다.
- Pragmatist: 현재 규모에서는 파일 분해보다 빠른 기능 개선이 더 큰 가치입니다.
- Skeptic: 문서가 코드와 분리되면 금방 틀어지므로 근거 경로와 테스트 기준을 항상 붙여야 합니다.
- Synthesis: 지금은 단순 구조 유지, 기능 증가 시점에 화면 로직부터 모듈화하는 점진 전환이 적절합니다.

## 좋은 점 / 제약 / 다음 진화

| 구분 | 내용 |
|---|---|
| 좋은 점 | 의존성이 작고, 흐름이 단순해 초급자가 전체를 빠르게 이해할 수 있음 |
| 제약 | 모듈 간 결합이 느슨하지 않으면 파일 분리만으로는 유지보수성이 충분히 오르지 않을 수 있음 |
| 다음 진화 | `public/index.html`의 API/렌더링/이벤트를 파일 분리하고, 공통 검증 로직을 모듈화 |

## 기술 부채와 확장 포인트

- 기술 부채 1: 프론트 스크립트 길이 증가
- 기술 부채 2: 타입 시스템 부재로 리팩터링 안전망 약함
- 기술 부채 3: 국제화/다통화 환산 정책이 아직 제한적

확장 포인트:

1. 프론트 로직 모듈화
2. 라우터의 엔드포인트별 파일 분리
3. 환율 테이블을 별도 서비스로 도입

## 실수하기 쉬운 포인트

- "단순하다"를 "아무 규칙 없다"로 오해하면 기능 확장 시 품질이 무너집니다.
- 트레이드오프를 문서화하지 않으면 후속 개발자가 왜 이렇게 만들었는지 재해석 비용이 커집니다.
- 저장소 추상화를 이미 썼다고 해서 스키마 변경 비용이 0이라고 착각하기 쉽습니다.
